# Rubric: PRD/需求文档评审

适用：产品需求文档（PRD）、功能规格说明书
不适用：代码改动、配置变更

## 完整性
- [ ] 问题陈述清晰：谁、什么问题、为什么现在要解决
- [ ] 用户故事完整：每条含角色、行为、价值（As a... I want... so that...）
- [ ] 范围边界明确：有「范围外」或「不包含」章节
- [ ] 未确定项标注 TBD（而非编造看似合理的需求填补）
- [ ] 若项目要求隔离开发环境，涉及代码改动的计划首步包含隔离环境创建

## 一致性
- [ ] 各决策项之间无自相矛盾（如 Decision A 依赖的功能被 Decision B 禁用）
- [ ] 接口/类型/数据模型引用均已定义，不只有名字无结构
- [ ] 优先级与问题陈述的紧迫性对齐

## 可行性
- [ ] 技术选型有理由，在当前架构下可实施
- [ ] 资源/时间估算有依据
- [ ] 外部依赖（API、服务、数据集）可用性已确认或标注假设

## 风险
- [ ] 已知风险已列出且有缓解措施
- [ ] 有回退方案或止损条件
- [ ] 兼容性影响（DB schema、API 版本、配置文件）已评估

## 可验证性
- [ ] 成功标准可衡量，非模糊描述
- [ ] 每条用户故事有独立验收条件
- [ ] 有测试策略，区分各级别（单元/集成/E2E）覆盖
